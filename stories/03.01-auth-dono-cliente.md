## Story

**Como** dono, **quero** fazer login com e-mail e senha para acessar o painel; **como** cliente, **quero** acessar minha área apenas pelo link que recebi (sem senha), **para que** o painel seja seguro e o cliente veja só seus serviços sem precisar cadastrar senha.

---

## Acceptance Criteria

1. Dono acessa tela de login (e-mail + senha); backend valida e emite sessão (JWT ou cookie) com role=dono.
2. Todas as rotas de gestão (clientes, serviços, status, notas) exigem autenticação dono; retornam 401 se não autenticado, 403 se role diferente.
3. Cliente acessa via URL com token (ex.: /acompanhar?token=xxx ou /acompanhar/xxx); backend valida token, associa a id_cliente e emite sessão com role=cliente e id_cliente.
4. Rotas /api/clientes/me/servicos e /api/clientes/me/servicos/:id exigem sessão cliente e retornam apenas dados do id_cliente da sessão; qualquer outra rota de escrita ou listagem geral retorna 403 para cliente.
5. Token do cliente: longo, aleatório, vinculado a id_cliente; pode ser único por cliente (um link mostra todos os serviços dele) conforme arquitetura.
6. Referência: ARQUITETURA-LOGICA-SITE-SI.md (seção 3.3 e 4.1, 4.2).

---

## Tasks / Subtasks

- [ ] Backend: modelo/armazenamento de usuário dono (email, hash de senha); geração e validação de JWT (ou sessão) com claim role=dono.
- [ ] Backend: geração e armazenamento de token de cliente (vinculado a id_cliente); validação de token e emissão de sessão com role=cliente e id_cliente.
- [ ] Backend: middleware de autenticação e autorização (extrair token/sessão; verificar role para cada rota).
- [ ] Backend: garantir que GET /api/clientes/me/* filtre sempre por id_cliente da sessão; demais rotas bloqueadas para role=cliente (403).
- [ ] Frontend (painel dono): tela de login (e-mail, senha); envio de credenciais; armazenamento de token/sessão; redirecionamento se não autenticado.
- [ ] Frontend (cliente): rota pública /acompanhar/:token ou /acompanhar?token=; troca de token por sessão; redirecionamento para lista "Meus serviços".
- [ ] Testes: login dono; acesso às APIs com token dono; acesso cliente com token válido (só suas rotas); 403 ao usar token cliente em rotas dono.

## Dev Notes

- Não implementar "esqueci senha" no MVP se não estiver no escopo; deixar link na tela para fase futura.
- Token cliente: pode ser armazenado em tabela (token_hash, id_cliente, criado_em) ou derivado de id_cliente + secret; definir estratégia segura com @architect se necessário.
- UX: TELAS-E-FLUXOS-SITE-SI.md (Login dono; Acesso por link + token).

## Testing

- Integração: login dono retorna token; requests com token dono acessam APIs; token cliente acessa apenas me/servicos; 401/403 corretos.
- Segurança: token cliente não acessa dados de outro cliente (inserir id_cliente em outro token e garantir 404/403).
